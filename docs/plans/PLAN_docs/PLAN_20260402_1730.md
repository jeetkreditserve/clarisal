## 🔒 Security Fixes

- [ ] **Stop the Control Tower (CT) support role from seeing employee private data** — The CT support dashboard currently returns government IDs (PAN, Aadhaar), bank account details, home address, and family member records when looking up any employee. - [ ] **Remove the ability for CT users to edit employee records** — A CT user can currently modify any employee record at any company in the system. CT is a support role and should only be able to view a sanitised snapshot, never edit. *(File: `backend/apps/organisations/views.py:321`)*

- [ ] **Make the app refuse to start if the data encryption key is not properly configured** — If the `FIELD_ENCRYPTION_KEY` environment variable is missing or set to the placeholder value, the system silently uses a fallback key. If the main app secret key is ever rotated, all PAN numbers, Aadhaar numbers, and bank account details stored in the database will become permanently unreadable. Add a startup check that raises an error if the encryption key is not set correctly. *(File: `backend/clarisal/settings/base.py:184`)*

- [ ] **Remove the default password from the setup script and stop printing credentials to the terminal** — The script that creates the initial Control Tower admin account uses a fallback password (`change-me-in-production`) and prints the resulting credentials to the terminal. - [ ] **Fix the CT employee pop-up in the frontend to no longer display phone numbers and home addresses** — The pop-up modal in the Control Tower org view says "Read-only support visibility" but still shows contact and address fields that the CT role is not supposed to see. *(File: `frontend/src/pages/ct/OrganisationDetailPage.tsx`)*

- [ ] **Validate the actual content of uploaded files, not just the filename** — When employees upload documents, the system checks only that the filename ends in `.pdf`, `.png`, or `.jpg`. - [ ] **Create separate, role-specific API response formats for CT, Org Admin, and Employee** — Right now CT and Org Admin share the same employee data format, which is why CT accidentally sees PII. ## 💸 Payroll Calculation Fixes (Indian Statutory Compliance)

The payroll engine currently produces mathematically incorrect results for any Indian company. These fixes address the specific bugs found during the audit.

- [ ] **Add the 4% Health & Education Cess on top of the income tax** — Every income tax computation in India requires an additional 4% cess on the tax amount. This is currently missing entirely, which means every employee is under-taxed. *(File: `backend/apps/payroll/services.py:117`)*

- [ ] **Apply the ₹75,000 standard deduction before calculating taxable income** — Indian tax law (new regime, FY2024–25 onwards) allows a flat ₹75,000 deduction from gross salary before tax is computed. Not applying this means employees are overtaxed by approximately ₹625/month. *(File: `backend/apps/payroll/services.py:480`)*

- [ ] **Auto-calculate PF (Provident Fund) as 12% of basic salary** — PF is currently stored as a fixed rupee amount (always ₹1,800). The correct rule is 12% of basic salary (capped at ₹1,800 only when basic is ≤ ₹15,000). An employee with ₹80,000 basic should contribute ₹9,600 to PF, not ₹1,800. *(File: `backend/apps/payroll/services.py`)*

- [ ] **Add ESI (Employee State Insurance) calculation** — ESI applies to all employees whose gross salary is ₹21,000/month or less. The employee contributes 0.75% and the employer contributes 3.25% of gross salary. This entire component is absent from the payroll engine. *(File: `backend/apps/payroll/services.py`)*

- [ ] **Add Professional Tax (PT) deduction** — PT is a state-mandated deduction that varies by state (e.g. Maharashtra: up to ₹200/month for salaries above ₹10,000). PT rules should be stored per office location's state and applied automatically during payroll. *(File: `backend/apps/payroll/services.py`)*

- [ ] **Calculate pro-rated salary for employees joining or leaving mid-month** — A person who joins on April 15 currently receives the full month's salary. The correct calculation is: (number of days worked / total days in month) × monthly salary. Same logic applies for exit month. *(File: `backend/apps/payroll/services.py:456`)*

- [ ] **Deduct salary for Loss of Pay (LOP) leave** — When an employee takes unpaid (LOP) leave, their pay should be reduced proportionally. The payroll calculation currently never reads leave request data, so LOP leaves have no effect on the payslip. *(File: `backend/apps/payroll/services.py:428`)*

- [ ] **Support old vs. new income tax regime selection per employee** — India has two income tax regimes (old and new) with different slabs and deduction rules. Employees choose which applies to them. Currently there is only one fixed tax slab structure with no regime distinction. *(File: `backend/apps/payroll/services.py`)*

- [ ] **Calculate the fiscal year dynamically instead of using a hardcoded string** — The current code contains `DEFAULT_FISCAL_YEAR = '2026-2027'` hardcoded. When April 2027 arrives, newly set up organisations will silently use the wrong year's tax master. *(File: `backend/apps/payroll/services.py:36`)*

## ⚙️ Architecture & Performance Fixes

- [ ] **Move payroll calculation to a background job** — The payroll calculation currently runs synchronously during an HTTP web request. For an organisation with 200 employees, this will take 2–5 seconds; for 500+ employees it will time out the web server. Move it to a Celery background task (the infrastructure already exists) and return a job status that the frontend can poll. *(File: `backend/apps/payroll/views.py:240`)*

- [ ] **Replace the leave days Python loop with a single database query** — When calculating an employee's leave balance, the code loads every leave request record into memory and sums them in Python. Replace this with a single SQL aggregate query (`SUM`), which is orders of magnitude faster at scale. *(File: `backend/apps/timeoff/services.py:292–314`)*

- [ ] **Remove the payroll setup side-effect from the GET (list) endpoint** — Every time an admin loads the tax slab list page, the server silently creates or modifies payroll setup records behind the scenes. A GET request (loading a page) should never cause data to be written. Move the setup call to a dedicated onboarding step. *(File: `backend/apps/payroll/views.py:98`)*

- [ ] **Add a database index to speed up leave balance queries** — The leave balance calculation queries the leave requests table by employee, leave type, status, and dates. Without an index on these columns, the database must scan the entire table every time. Adding a composite index will significantly speed this up as data grows. *(File: `backend/apps/timeoff/models.py`)*

- [ ] **Fix the leave balance calculation to use a fixed "as of" date** — An employee's leave balance currently changes silently every day because the calculation uses "today's date" internally. This makes it impossible to calculate what the balance was on a past date (e.g. "what was the balance at the end of March?"). Pass a specific date through consistently. *(File: `backend/apps/timeoff/services.py:248`)*

- [ ] **Split the payroll summary endpoint into separate paginated APIs** — One API call currently fetches all tax slabs, all salary templates, all employee assignments, and all historical payroll runs at once. For an org with 200 employees and 2 years of history, this returns thousands of nested records. Split into separate paginated endpoints. *(File: `backend/apps/payroll/views.py:71`)*

- [ ] **Add API versioning (`/api/v1/`)** — All APIs are currently at `/api/*` with no version prefix. Once the attendance or payroll modules mature, changing the API will break existing clients. Introduce versioning before adding more surface area. *(File: `backend/clarisal/urls.py`)*

- [ ] **Standardize API error response format** — Different API endpoints return errors in different shapes. Some return `{'error': 'message'}`, others return DRF validation errors with field names. Define one consistent error format with a code, message, and optional field-level details so the frontend can handle all errors uniformly. *(Multiple view files)*

- [ ] **Extract duplicate helper functions into one shared location** — The same two helper functions (`_get_admin_organisation` and `_get_employee`) are copy-pasted across 6+ different files. Any bug fix must be applied to all copies. Move them to a single shared module. *(Files: `backend/apps/payroll/views.py`, `backend/apps/communications/views.py`, `backend/apps/approvals/views.py`, `backend/apps/employees/views.py`, and others)*

- [ ] **Delete the misspelled stale package folder `backend/calrisal/`** — A folder with a typo in its name (`calrisal` instead of `clarisal`) exists as a residue from an earlier error. It creates structural confusion and can mislead development tools. *(File: `backend/calrisal/`)*
