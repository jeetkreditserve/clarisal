# Combined Remediation Plan For The HRMS Audit Findings

## What This Plan Is For
This plan is for addressing the issues raised in these two audit reports:

- [HRMS_AUDIT_REPORT_20260402_1711.md](/home/jeet-padhya/PycharmProjects/clarisal/docs/HRMS_AUDIT_REPORT_20260402_1711.md)
- [HRMS_AUDIT_REPORT_20260402_1716.md](/home/jeet-padhya/PycharmProjects/clarisal/docs/HRMS_AUDIT_REPORT_20260402_1716.md)

The goal is to turn those audit findings into a **working action checklist** that can be used during implementation. As each task is finished, it should be marked as done.

This plan is written for a non-technical reader first. It explains:

- what must be fixed immediately
- what can wait for the next sprint or two
- what belongs in the longer roadmap
- how to track progress in a simple checkbox format

## How To Use This Plan
- [ ] Leave a task as `- [ ]` while it is not complete
- [ ] Change a task to `- [x]` when it is finished
- [ ] If only part of a task is done, leave it unchecked and add a short note under it
- [ ] Do not mark a full phase complete until every item inside it is done

## Definition Of Done
This audit-remediation effort is only considered complete when all of the following are true:

- [ ] The most urgent privacy and security issues are fixed
- [ ] The Control Tower role is restricted to the right level of visibility
- [ ] Attendance is either built properly or clearly removed from any misleading partial UI/workflow exposure
- [ ] Payroll is safe enough to use for real-world operation within the supported scope
- [ ] The product no longer gives false confidence about missing features
- [ ] The user experience is improved for the most important admin and employee journeys
- [ ] The biggest missing HRMS areas have clear next steps and owners
- [ ] Progress has been tracked in this checklist as work was completed

## Phase 1 - Immediate Safety And Access Fixes
Done means: the product is no longer exposing sensitive data in ways the audit marked as critical.

### 1.1 Lock Down Control Tower Access To Employee Data
- [x] Stop Control Tower from seeing employee details it should not see
- [x] Restrict Control Tower to a sanitised employee view only
- [ ] Remove any Control Tower access to:
  - [x] home address
  - [x] PAN details
  - [x] Aadhaar details
  - [x] bank details
  - [x] family information
  - [x] emergency contacts
  - [ ] salary details
- [x] Remove Control Tower’s ability to edit employee records
- [x] Re-check every Control Tower employee API response after the fix
- [x] Re-check every Control Tower employee UI screen after the fix
- [x] Add automated checks so this privacy boundary does not break again later

### 1.2 Rotate Exposed Credentials And Clean Up Secret Handling
- [ ] Rotate AWS credentials if they are real and still active
- [ ] Rotate ZeptoMail credentials if they are real and still active
- [ ] Review the `.env` file and confirm whether any live secrets are present
- [ ] Remove any unsafe example credentials from documentation or scripts
- [x] Stop printing or echoing credentials in any setup or seed command
- [ ] Confirm no secrets remain in plain form where they should not be

### 1.3 Harden Sensitive Data Protection
- [ ] Make field-encryption configuration mandatory in production
- [ ] Confirm encrypted fields can still be read safely after the change
- [ ] Add a clear operational note on how encryption keys are managed
- [ ] Confirm sensitive downloads are still protected and time-limited
- [ ] Add better tracking for who requested access to sensitive documents

## Phase 2 - Clarify Product Scope So The Product Does Not Mislead Users
Done means: the product no longer looks more complete than it actually is.

### 2.1 Fix Misleading Attendance Signals
- [ ] Review all UI and workflow references to attendance regularization
- [ ] Remove or hide attendance-related workflow options if they do not lead to a real working feature
- [ ] Review nav menus and pages so the product does not imply a working attendance module where one does not exist
- [ ] Add plain messaging where needed so admins understand attendance is not yet available

### 2.2 Fix Misleading Payroll Signals
- [ ] Review all payroll page copy and labels
- [ ] Remove wording that implies full statutory payroll support if it does not exist yet
- [ ] Add a clear warning or limited-scope label if payroll is still in prototype form
- [ ] Make sure users are not encouraged to use payroll for live statutory processing until it is truly ready

### 2.3 Fix Misleading Role Boundaries
- [ ] Review CT screens and copy so they present CT as a support/super-admin layer with the correct limits
- [ ] Review Org Admin screens so they do not imply missing reporting or attendance features exist
- [ ] Review Employee screens so they do not imply attendance and tax self-service are available when they are not

## Phase 4 - Repair Payroll So It Is Safe For Real Use
Done means: payroll is either narrowed to a safe scope or improved enough that it no longer produces obviously incorrect outcomes.

### 4.1 Fix Payroll Calculation Correctness
- [ ] Review how tax is calculated today
- [ ] Add missing India payroll logic called out in the audits:
  - [ ] ESI
  - [ ] Professional Tax
  - [ ] TDS depth
  - [ ] cess
  - [ ] surcharge where applicable
  - [ ] standard deduction where applicable
- [ ] Fix PF so it is not treated as a flat manual number where it should be formula-driven
- [ ] Fix pro-rated salary for joining month
- [ ] Fix pro-rated salary for exit month
- [ ] Connect approved LOP leave to payroll deductions
- [ ] Confirm employer contributions are handled correctly for reporting even if they do not affect take-home pay

### 4.2 Fix Payroll Scope Gaps
- [ ] Add a proper payslip output instead of plain text only
- [ ] Add payroll reports needed for operational use
- [ ] Add Form 16 support if the product is meant to support real India payroll
- [ ] Add investment declaration handling if TDS is expected to be accurate
- [ ] Add arrears and revision handling if salary changes are meant to be operationally safe
- [ ] Add full & final settlement if employee exits are meant to be supported properly

### 4.3 Make Payroll Safer To Operate
- [ ] Add confirmation before payroll finalization
- [ ] Add better exception handling during payroll processing
- [ ] Make it easy to understand why a payroll run failed
- [ ] Separate payroll setup from payroll execution so one page is not overloaded
- [ ] Consider moving heavy payroll processing out of user-facing request time

### 4.4 Payroll Positioning Check
- [ ] Decide whether payroll should remain prototype-only for now
- [ ] If yes, label it clearly as limited-scope and block production-style use
- [ ] If no, do not go live until the calculation and compliance gaps above are complete
